## Decisions log

- 2026-07-15: Development tooling uses exact-pinned PHPCS 3.13.5, WPCS 3.3.0, PHPCompatibilityWP 2.1.8, PHPUnit 9.6.35, and PHPUnit Polyfills 4.0.0. The polyfills package is part of the WordPress test-suite scaffolding needed to run the supported PHPUnit versions across PHP releases.
- 2026-07-15: Storage uses WordPress post meta only (`_wpfaq_faqs`, `_wpfaq_display_location`, `_wpfaq_custom_tabs`); no custom tables in v1 to stay simple. A custom table would require `dbDelta` with a versioned `wpfaq_db_version` option and an upgrade routine, and must be recorded here before adoption.
- 2026-07-15: Prefix set to `wpfaq_` (functions/hooks), `WPFAQ_` (constants/classes); text domain `woo-product-faq`.
- 2026-07-15: Frontend accordion is vanilla JS + CSS (no framework, no build step); admin reorder reuses WordPress-bundled jQuery UI Sortable to avoid a new dependency.
- 2026-07-15: Only four classes allowed without approval: `WPFAQ_Plugin`, `WPFAQ_Admin`, `WPFAQ_Frontend`, `WPFAQ_Tabs`. Data access lives in prefixed functions in `wpfaq-functions.php`.
- 2026-07-15: FAQs attach to the parent product only; variations do not get separate FAQs in v1.
- 2026-07-15: FAQ JSON-LD structured data and a shortcode/block are deferred to backlog, not v1.
- 2026-07-17: `phpcs.xml.dist` gained a `custom_test_classes` property (`WordPress.Files.FileName` rule) listing `WC_Unit_Test_Case`, so PHPUnit test files using WooCommerce's base test case are exempted from the `class-*` file-naming rule the same way core `WP_UnitTestCase` files already are. This is a lint-config addition, not a new runtime/dev dependency; flagged here per the "no new dependency without approval" rule even though nothing was installed.
- 2026-07-17: Custom-tab empty-row rule intentionally differs from FAQ rows: a custom tab is dropped if EITHER the title OR the content is empty (per PRD's "Extra custom product tabs" success criteria), whereas an FAQ row is only dropped if BOTH question and answer are blank (per PRD's "Product FAQ editor" success criteria). Implemented as two separate sanitize methods in `WPFAQ_Admin` (`sanitize_faq_rows`, `sanitize_custom_tabs`) rather than one shared helper, since the emptiness rule differs and rules.md's "rule of three" doesn't justify an abstraction yet.
- 2026-07-17: Uninstall cleanup uses `delete_post_meta_by_key()` (a WP core API, not raw SQL) once per meta key rather than looping over a product query; simpler and matches rules.md's "no `$wpdb` queries" constraint.

## Verification gaps (flagged, not silently absorbed)

- `wp-env start` (Docker-based WP+WooCommerce environment) was attempted twice per the task instructions and failed both times in this sandbox: first with a broken `@wordpress/env` npx cache (`Cannot find module 'signal-exit'`), and second after clearing that cache, the command hung with no output until the shell timeout killed it (Docker image pulls likely blocked or too slow in this environment). Per the fallback instruction, further wp-env attempts were abandoned after these two tries.
- As a substitute, a real WordPress + WooCommerce PHPUnit integration environment was built by hand: `bin/install-wp-tests.sh` against a local `mariadb:10.6` Docker container, downloading real WordPress core (6.7) and WooCommerce 10.9.4 test fixtures. This proved the plugin bootstraps cleanly against real WordPress + WooCommerce with **zero fatals** (`tests/test-bootstrap.php`, a plain `WP_UnitTestCase` test, and `tests/test-uninstall.php`, which exercises the real `uninstall.php` file end-to-end, both passed against this real stack).
- The FAQ/custom-tab/tabs feature tests (`tests/test-save-faqs.php`, `tests/test-get-faqs.php`, `tests/test-tabs.php`) extend `WC_Unit_Test_Case`, which is WooCommerce's own test base class. It is **not** shipped in the distributed WooCommerce plugin zip — it lives in WooCommerce's separate dev/test framework (`tests/legacy/framework`, `tests/Tools`, WC's own composer-installed DI container, etc.) in the `woocommerce/woocommerce` GitHub repo. Pulling in that entire framework would mean vendoring a large chunk of WooCommerce's dev tooling, which reads as a new dev dependency requiring approval per rules.md ("no new runtime or dev dependency without explicit approval"). Two attempts were made to get a lighter version working before stopping per the "two failed attempts" rule; these three test files are therefore verified by `php -l` + PHPCS (both clean) and manual logic review only, not by an actual PHPUnit run, in this sandbox. They should run cleanly in a real dev environment with the full WooCommerce dev checkout (e.g. via `wp scaffold plugin-tests` against a WooCommerce dev install, as `docs/testing.md` assumes).
- Beyond PHPUnit, a real end-to-end smoke test was run: WordPress 7.0.1 + WooCommerce 10.9.4 installed via WP-CLI on the built-in PHP dev server, with this plugin activated via a symlink. Confirmed live and working: plugin + WooCommerce activate with no fatals; the admin product-edit screen renders the FAQ panel, the display-location select (correct `selected` value), populated FAQ rows, populated custom-tab rows, and correctly-hidden empty-state hints; the frontend product page (default `tab` location) renders the FAQ tab and the custom tab with correct escaped content and no PHP warnings/notices in the server log.
- One rendering path could **not** be confirmed live in this sandbox: the `after_summary` display location (`WPFAQ_Frontend::render_after_summary()`, hooked on `woocommerce_after_single_product_summary`). On the built-in PHP dev server with plain (non-pretty) permalinks, `do_action( 'woocommerce_after_single_product_summary' )` never fired for a product requested as `/?p=ID&post_type=product` — confirmed with an unconditional throwaway `add_action` marker (never printed) and by checking that `template_include` itself never ran for that exact request (it did run correctly for the homepage). This reproduced identically on both a block theme (Twenty Twenty-Four) and a classic theme (Storefront, which does declare `add_theme_support( 'woocommerce' )`), so it is not a template-hook-registration bug in this plugin. Calling `WPFAQ_Frontend::render_after_summary()` directly in isolation (with the real product's saved meta) produces exactly correct, well-formed accordion markup, so the plugin's own logic is verified correct at the unit level. The most likely explanation is an interaction specific to the PHP built-in dev server plus plain permalinks in this environment (possibly a WooCommerce block-based single-product template path bypassing the classic template's `template_include` filter entirely, which needs a real Apache/Nginx + pretty-permalinks setup to rule in or out). Recommend re-running this specific check (`docs/phases.md` Phase 2/3 manual checklist: "Switch to 'After product summary', save: FAQ tab gone, accordion under summary") on a real server with pretty permalinks before shipping; flagging rather than guessing further per the "stop after two failed attempts" rule.
